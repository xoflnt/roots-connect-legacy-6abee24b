**الهدف:** تحويل منصة عائلية من ٠ إيرادات إلى نموذج أعمال مستدام

## المقدمة التنفيذية

ناسبي هي منصة SaaS عربية متعددة المستأجرين لإدارة أشجار الأنساب العائلية. حالياً:
- **عائلتان مسجلتان:** الخنيني (٤١٠+ أفراد) + عائلة ثانية
- **الإيرادات:** صفر
- **نموذج التسعير:** استشاري (لا يوجد سعر ثابت — نقاش شخصي حسب الحجم والحاجة)
- **الفريق:** عبدالله (غير مطور، طالب محاسبة) + مطور مستقل (أنا)
- **الهيكل التقني:** React + Vite + Supabase + PWA

**الرؤية:** منتج وراثة عربي مستدام، لا سباق نحو مليار دولار. التركيز على القيمة الحقيقية للعائلات السعودية والخليجية.

**النقطة المركزية:** الثقة. العائلات تسلّم بيانات عائلية حساسة جداً (أسماء، هويات، علاقات زواج). لا توجد منصة أخرى عربية متخصصة تفهم الحساسية الثقافية.

     ```sql
     -- Family billing profiles
     CREATE TABLE family_billing (
       id UUID PRIMARY KEY,
       family_id TEXT UNIQUE NOT NULL,
       monthly_price_sar INT NOT NULL,      -- السعر المتفق عليه (مثال: 500-2000 ريال)
       status TEXT DEFAULT 'active',        -- active, trial, paused, archived
       trial_ends_at TIMESTAMP,
       created_at TIMESTAMP DEFAULT now()
     );

     -- Monthly invoices
     CREATE TABLE invoices (
       id UUID PRIMARY KEY,
       family_id TEXT NOT NULL REFERENCES family_billing(family_id),
       amount_sar INT NOT NULL,
       status TEXT DEFAULT 'pending',       -- pending, paid, overdue, cancelled
       issued_at TIMESTAMP DEFAULT now(),
       due_date TIMESTAMP,
       paid_at TIMESTAMP,
       moyasar_transaction_id TEXT UNIQUE
     );

     -- Payment logs
     CREATE TABLE payment_logs (
       id UUID PRIMARY KEY,
       invoice_id UUID REFERENCES invoices(id),
       amount_sar INT NOT NULL,
       status TEXT,                         -- success, failed, refunded
       error_message TEXT,
       moyasar_response JSONB,
       attempted_at TIMESTAMP DEFAULT now()
     );
     ```

### **الأسبوع الثاني (٨-١٤ أبريل): بناء تكامل Moyasar وخط أنابيب الفواتير**

**المهام:**

1. **كود التكامل مع Moyasar (يوم الاثنين - الأربعاء)**
   - Edge function جديد: `moyasar-payment`
   ```typescript
   // supabase/functions/moyasar-payment/index.ts
   import { serve } from "https://deno.land/std@0.168.0/http/server.ts";
   import { createClient } from "https://esm.sh/@supabase/supabase-js";

   const MOYASAR_API_KEY = Deno.env.get("MOYASAR_API_KEY")!;
   const MOYASAR_BASE_URL = "https://api.moyasar.com/v1";

   serve(async (req) => {
     if (req.method !== "POST") return new Response("Method not allowed", { status: 405 });

     const { invoiceId, amountSAR, callbackUrl } = await req.json();

     // Validate invoice exists
     const supabase = createClient(
       Deno.env.get("SUPABASE_URL")!,
       Deno.env.get("SUPABASE_SERVICE_ROLE_KEY")!
     );

     const { data: invoice } = await supabase
       .from("invoices")
       .select("*")
       .eq("id", invoiceId)
       .single();

     if (!invoice) {
       return new Response(
         JSON.stringify({ error: "Invoice not found" }),
         { status: 404 }
       );
     }

     try {
       // Create Moyasar payment source
       const response = await fetch(`${MOYASAR_BASE_URL}/payments`, {
         method: "POST",
         headers: {
           "Authorization": `Basic ${btoa(MOYASAR_API_KEY + ":")}`,
           "Content-Type": "application/json",
         },
         body: JSON.stringify({
           amount: amountSAR * 100,  // Convert to halalas
           currency: "SAR",
           description: `فاتورة ناسبي #${invoiceId.substring(0, 8)}`,
           metadata: { invoiceId, familyId: invoice.family_id },
           source: {
             type: "redirect",
             redirect: callbackUrl,
           },
         }),
       });

       const payment = await response.json();

       // Store payment log
       await supabase
         .from("payment_logs")
         .insert({
           invoice_id: invoiceId,
           amount_sar: amountSAR,
           status: payment.status,
           moyasar_response: payment,
           moyasar_transaction_id: payment.id,
         });

       return new Response(
         JSON.stringify({
           paymentUrl: payment.source.redirect.url || payment.source.redirect_url
         }),
         { status: 200, headers: { "Content-Type": "application/json" } }
       );
     } catch (error) {
       console.error("Moyasar error:", error);
       return new Response(
         JSON.stringify({ error: error.message }),
         { status: 500 }
       );
     }
   });
   ```

   ```typescript
   // src/utils/excelImporter.ts
   import * as XLSX from "xlsx";

   export interface ImportRow {
     name: string;
     gender: "M" | "F";
     phone?: string;
     birthYear?: string;
     father?: string;      // Name lookup
     mother?: string;
     spouses?: string;     // Comma-separated
     notes?: string;
   }

   export async function parseExcelFile(file: File): Promise<ImportRow[]> {
     const workbook = XLSX.read(await file.arrayBuffer());
     const sheet = workbook.Sheets[workbook.SheetNames[0]];
     const rows = XLSX.utils.sheet_to_json(sheet, { defval: "" });

     return rows.map((row: any) => ({
       name: row["الاسم العربي"] || row.name || "",
       gender: (row.الجنس || row.gender || "M")[0].toUpperCase(),
       phone: row["رقم الهاتف"] || row.phone || undefined,
       birthYear: row["سنة الميلاد الهجرية"] || row.birthYear,
       father: row.الأب || row.father,
       mother: row.الأم || row.mother,
       spouses: row.الزوجات || row.spouses,
       notes: row.ملاحظات || row.notes,
     }));
   }

   export async function matchFatherIds(
     rows: ImportRow[],
     existingMembers: FamilyMember[]
   ): Promise<{ memberId: string; fatherId?: string }[]> {
     // استخدام fuzzy matching على الأسماء
     const nameMap = new Map(existingMembers.map(m => [m.name, m.id]));

     return rows.map(row => ({
       memberId: row.name,
       fatherId: nameMap.get(row.father || "") || undefined,
     }));
   }
   ```

## الجزء ب: العائق الأوحد (الجبل الذي يجب تسلقه)

### **العائق: بناء الثقة والمصداقية في سوق جديد تماماً**

**التفاصيل:**

لا توجد منصة SaaS عربية متخصصة في إدارة أشجار الأنساب. المنافسة العالمية (Ancestry.com، FamilySearch) محايدة ثقافياً ولا تفهم الحساسيات السعودية حول البيانات العائلية (من يمكنه رؤية المعلومات الحساسة؟ هل يجب إخفاء أسماء الإناث؟ كيف نتعامل مع زيجات متعددة؟).

**لماذا هذا هو العائق:**
1. العائلات السعودية الكبيرة لن تسلّم بيانات حساسة لمنصة غير موثوقة
2. لا يوجد "إثبات اجتماعي" (social proof) — أول عائلة تختارنا تقامر برقتها
3. التنافس مع المنصات العالمية على الثقة صعب جداً

**الحل:**

1. **استثمار الثقة الشخصية (الأسابيع ١-٨)**
   - عبدالله (صاحب المنتج) يجب أن يكون وجهاً شخصياً
   - اتصالات مباشرة مع رؤساء العائلات الكبيرة
   - نقاشات حول الخصوصية والأمان (ليس كـ "ميزات تقنية" بل كـ "التزامات")

2. **شهادات ودية من العائلات الأولى (الأسابيع ٥-٨)**
   - بعد 3 أشهر من استخدام الخنيني: حوار مكتوب أو فيديو قصير
   - الموضوع: "كيف ساعدتنا ناسبي في توحيد تاريخ عائلتنا"
   - نشر على موقع ناسبي الرئيسي

3. **محتوى تعليمي بالعربية عن الخصوصية (الأسبوع ٩)**
   - مقالة طويلة: "كيف نحمي بيانات عائلتك: معايير أمان ناسبي"
   - محتوى يشرح:
     - تشفير البيانات في الراحة والنقل (Supabase encryption)
     - سياسة الوصول (من يرى ماذا)
     - عدم مشاركة البيانات مع طرف ثالث أبداً
     - امتثال GDPR و النظام السعودي للبيانات الشخصية

4. **برنامج "الراعي الموثوق" (الأسابيع ١٠-١٢)**
   - اختيار 3 عائلات كـ "عائلات الراعي" (founding customers)
   - استثمار وقت شخصي كبير جداً في نجاحهم
   - هدية: خصم 50% السنة الأولى
   - مقابل: تصريح باستخدام قصتهم في التسويق

**الخطوة التالية المحددة:** حدد 3 عائلات سعودية معروفة (الخنيني + عائلتان أخريان) وابدأ محادثات شخصية بخصوص "لماذا أنتم تستحقون منصة آمنة خاصة."

| **معدل النمو** | 50% YoY في الحالة المتفائلة | شهر: 1 عائلة جديدة |

### **المقاييس الحرجة للمراقبة:**

| المقياس | الهدف (شهر 12) | الأهمية |
|--------|--------|--------|
| **MRR (شهري متكرر)** | ٨,٠٠٠ ريال | حد الأدنى للاستدامة |
| **CAC (تكلفة اكتساب العميل)** | < ٢,٠٠٠ ريال | إذا > ٢,٠٠٠ فالنموذج معطل |
| **LTV (القيمة الحياتية)** | > ٣٦,٠٠٠ ريال | LTV/CAC يجب أن يكون > 3 |
| **Churn rate** | < 5% شهري | إذا > 10% فنحن نفقد العملاء |
| **NRR (Net Revenue Retention)** | > 100% | يعني عائلات موجودة تشتري ميزات إضافية |

### **كيف نصل إلى السيناريو المتفائل؟**

**الخطوة التالية المحددة:** تتبع MRR كل أسبوع. إذا كنا أقل من السيناريو المتفائل بـ 40% بحلول الشهر 4، نراجع الاستراتيجية.

## الخلاصة التنفيذية: الأولويات للـ 90 يوم القادمة

### **الأعلى أهمية:**

1. **Moyasar integration + invoicing** (أسابيع 1-2)
   - بدون نظام دفع، لا توجد إيرادات
   - الخطوة التالية المحددة: اختبار عملية دفع كاملة بـ 100 ريال تجريبياً يوم 15 أبريل

2. **تحويل الخنيني إلى عميل دافع** (الأسبوع 4)
   - أول إيرادات = أول اختبار حقيقي للنموذج
   - الخطوة التالية المحددة: اجتماع بيع مع عبدالله يوم 22 أبريل

3. **البحث والتواصل مع 5 عائلات** (الأسابيع 5-8)
   - لا نمو بدون مبيعات نشطة
   - الخطوة التالية المحددة: قائمة من 10 عائلات محتملة بحلول 13 مايو

### **المهم لكن ليس عاجلاً:**

4. Excel/GEDCOM import (الأسابيع 3، 9-10)
5. White-label (الأسبوع 10)
6. شراكات مؤسسية (الأسبوع 11)

### **المؤشرات الحمراء التي تتطلب تغيير المسار:**

- **إذا:** لم تحصل على أي عائلة جديدة بحلول الأسبوع 8 → غيّر رسالة التسويق أو السعر
- **إذا:** MRR أقل من ١,٠٠٠ ريال بحلول الشهر 6 → اطلب تمويلاً أو قلل الرواتب
- **إذا:** Churn rate > 20% → المنتج نفسه مكسور أو التوقعات غير واقعية
